Overview

The engagement APIs allow a tenant to use a third party (an engagement vendor) to perform some kind of engagement action on a candidate or a job application, where that interaction results in data that helps predict the candidate's suitability for employment.

An engagement might be anywhere along the scale from attract to hire, and might or might not involve the candidate themselves, for example:

  • undertaking a skills assessment
  • carrying out reference checks
  • cultural/fit analysis
  • carrying out medical checks or drug tests
  • gamification interaction
  • probity check
  • HR-XML style assessment

The API design is informed by the work of the HR-XML consortium, but is also suitable for very lightweight engagements.

The engagement vendor delivers their engagement functionality as an app (an engagement app) that the tenant can install.

Engagement types

The tenant requests an engagement by selecting one of the engagement types that the vendor offers.

Some engagement vendors may provide highly customised engagement types, tailored to each individual customer's needs (e.g., "Acme finance graduate assessment"), others just a single, generally applicable screen (e.g. "MS Word proficiency").

The engagement app may update/complete the engagement through APIs, or an app is provided that lets zero-code vendors update the engagement via screens.

Supporting apps

Engagement hub app

As well as a engagement app, the tenant must also install a single "engagement hub" app, which provides centralized engagement management, and is separate from any single engagement app.

The engagement hub app:

  • stores details of all engagement, including status, summary and detailed results (including scores), links and actual attachments
  • stores details of all of the tenant's installed engagement apps - for example, the "Microsoft Office proficiency assessment app"
  • stores a tenant-specific list of engagement types within each engagement app - for example "MS Word proficiency - entry level", "MS Excel proficiency - guru"
  • allows the tenant to overwrite the title on each engagement type (for example, to rename "MS Excel proficiency - guru" to "Excel expert"
  • allows the tenant to specify a view against each engagement type - e.g. to make available the candidate's completed "FSA disclosure" document when a "Finance team probity check" is performed.
  • can optionally copy down the set of default engagement types whenever a new tenant installs the engagement app (useful for engagement apps that have fixed, generally applicable purpose/volume engagement types; and/or
  • provides APIs so the engagement vendor can manage engagement types directly for any given tenant (useful when the tenant has a customised set of engagement types)
  • stores the completed engagements
  • securely mediates engagement app access to views of information about the candidate/application, and prevents engagement apps from probing candidates they are not authorized to access
  • detects and securely tracks changes in view data (e.g. in the ATS, an additional document is uploaded against the candidate while the engagement is in progress)
  • prevents engagements being ordered when an existing report is available within the expiry time

Engagement vendor app

A developer who is an engagement vendor must maintain a tenant of their own, and must install the engagement vendor app. Covered more in detail below.

Engagement vendor UI app

An engagement vendor will also usually install the engagementvendorui app which provides UI to:

  • View engagement apps
  • View tenants that have installed each app
  • Manually manage engagement types for each install
  • Manually complete a engagement (for engagement vendors with no technology of their own - others will complete engagements in their own platform and use APIs to post the results to the engagement hub)

In theory an engagement vendor can accomplish all of the above through APIs so the engagement vendor app is optional.

Differences from HR-XML

The HR-XML ScreeningResultTypeCode is not currently supported, as some of its possible values ('Result'|'SummaryResult'|'Error'|'Message'|'Status'|'ManagementReport'|'MultipleResults') clash with ScreeningStatusCode values (e.g. Error), or are not suited to microservices (e.g. MultipleResults).

The ScreeningResultCode is not currently supported, as some of its possible values ('Pass'|'Fail'|'Review'|'RecordFound'|'NoRecordFound'|'Clear'|'UnableToContact'| 'UnableToFulfill'|'UnableToVerify'|'Positive'|'Negative') are ambiguous (HR-XML notes that *this set of codes usually would need to be interpretated based on the type of screening and pre-existing understanding between service provider and customer on criteria governing the application of the codes*), e.g. Pass vs Positive.

Details

Candidate applies for a job and appears in the ATS (jobly).

The recruiter clicks an ATS menu item to "start an engagement".

ATS user gets the list of possible engagement types

The ATS (jobly) asks the engagement hub app for the list of available engagement types, as served up by the sum of the installed engagement apps.

GET /deployedEngagementTypes Response:

A more sophisticated ATS might:

  • filter this list, to only present certain engagement apps/ screen types for certain jobs, according to some mappings held by the ATS, e.g. only call center candidates are asked to undertake this assessment.

Request an engagement

The ATS user chooses an engagement type and hits "go".

The ATS determines the candidate (or job application) that the engagement is for - in this case candidate 20887342.

The ATS calls:
POST /engagements/{engagementType}

Admin person sets up candidate view

Views are an ordered set of fields and attachments related to a candidate that will be pulled together for some purpose, e.g. being sent across to a third party who will be reference checking the candidate.

In this case, the view specifies what the person/machine performing the screen will see.

Engagement vendor uses UI to respond to an order

ATS integration

The ATS may decide to do any of the following:

  • automatic firing
  • manual firing
  • on response
    • waiting
    • moving to a new bucket
  • showing results summary in a table cell
  • reusing existing results
  • cancelling when declined

Engagementvendorui screens

Within their own tenant, the engagement vendor has installed the app "engagementvendorui".

They can use this to see how many engagements they are currently working on, and for which tenants.
